# 이론 정리

## 1. 왜 이 개념이 필요한가

기능이 늘어나면 저장만큼 조회 부담도 중요해집니다.
같은 데이터를 자주 읽는 상황에서 매번 DB를 조회하면 응답 속도와 DB 부하가 모두 부담이 될 수 있습니다.

이번 시퀀스는 게시글 단건 조회 앞에 Redis 캐시를 두고, cache hit, cache miss, TTL, stale data를 함께 이해하는 단계입니다.

## 2. 기존 방식의 한계

DB는 기준 데이터 저장소입니다.
하지만 자주 반복되는 조회까지 모두 DB가 처리하면 같은 데이터를 계속 읽는 비용이 커질 수 있습니다.

반대로 캐시만 믿으면 최신성이 흔들릴 수 있습니다.
DB는 바뀌었는데 캐시에는 예전 값이 남아 있는 stale data가 생길 수 있기 때문입니다.

## 3. 이번 시퀀스에서 선택한 접근

- 먼저 Redis 캐시에서 게시글을 찾습니다.
- 값이 있으면 cache hit로 바로 응답합니다.
- 값이 없으면 cache miss로 보고 DB를 조회합니다.
- DB 조회 결과를 Redis에 저장하고 TTL을 둡니다.
- 수정 이후 캐시를 어떻게 정리해야 하는지도 함께 생각합니다.

## 4. 핵심 개념

### Cache

자주 다시 쓰는 데이터를 더 빠르게 꺼내기 위해 잠깐 보관하는 보조 저장소입니다.
DB를 대체하는 저장소가 아니라 조회 부담을 줄이는 장치입니다.

### Redis

메모리 기반으로 값을 빠르게 저장하고 조회할 수 있는 저장소입니다.
이번 시퀀스에서는 가장 단순한 문자열 캐시 저장소로 사용합니다.

### Cache-aside

먼저 캐시를 보고, 값이 없으면 DB를 조회한 뒤 다시 캐시에 저장하는 패턴입니다.
기존 DB 조회 흐름 앞에 캐시 레이어를 얹기 좋은 입문 방식입니다.

### Cache hit / miss

hit는 캐시에 값이 있어 바로 응답하는 상황입니다.
miss는 캐시에 값이 없어 DB 조회로 이어지는 상황이며, 실패가 아니라 정상 흐름입니다.

### TTL과 stale data

TTL은 캐시 값이 살아 있는 시간을 정합니다.
하지만 TTL이 끝나기 전까지는 오래된 값이 남을 수 있으므로 수정/삭제 시점의 evict도 함께 생각해야 합니다.

## 5. 짧은 예제와 해설

이번 흐름은 `요청 -> 캐시 조회 -> miss면 DB 조회 -> 캐시 저장 -> 응답`입니다.
실습자는 `PostQueryService`에서 hit/miss 분기가 어디에 놓이는지, `PostCacheService`에서 key와 TTL이 어디에 모이는지 확인합니다.

수정 이후 캐시를 지우지 않으면 DB와 Redis가 서로 다른 값을 가질 수 있습니다.
그래서 조회 캐시를 붙일 때는 "언제 저장할 것인가"와 함께 "언제 지울 것인가"를 같이 설계해야 합니다.
